# Streamlit 最小回测页面设计

## 背景与目标

当前仓库是一个纯 Python 的量化研究脚手架，核心流程统一收口在 `src/quant_a/main.py` 的 `run_pipeline()`。本次目标是在尽量少改动现有结构的前提下，为该流程补一个可交互的 Web 页面，使用户可以：

- 点按钮触发回测
- 调整少量核心参数
- 在线查看指标、图表和订单结果
- 继续保留现有 `orders/` 与 `reports/` 文件输出

本次设计只覆盖第一版最小可用页面，不做产品化系统建设。

## 设计原则

1. **页面只做展示与触发，不承载策略逻辑。**
2. **保留现有 CLI 默认行为。**
3. **参数覆盖仅作用于本次运行，不写回配置文件。**
4. **优先复用现有流水线结果，不在页面重复计算。**
5. **明确排除产品化需求，避免过度设计。**

## 方案选择

### 方案 A：Streamlit 薄壳页（推荐）
新增一个单文件 Streamlit 页面入口，负责采集参数、触发 `run_pipeline()`、展示结果，并继续显示落盘文件路径。

**优点**
- 改动最小
- 学习和维护成本低
- 非常适合单人研究和轻量演示

**缺点**
- 后续如果要做多用户、异步任务、权限和复杂页面，扩展边界较快到来

### 方案 B：先拆服务层再接 Streamlit
先增加一层更明确的服务接口，再让页面调用。

**优点**
- 结构更规整
- 后续接 API 更顺

**缺点**
- 首版增加了不必要的工程化工作

### 方案 C：FastAPI + 前端
直接构建 API 与前端。

**优点**
- 扩展性最好

**缺点**
- 明显超出当前目标
- 与“最小接法”不符

**结论**：第一版采用方案 A。

## 第一版范围

### 必须支持
- 启动一个 Streamlit 页面
- 调整 4 个核心参数
- 点击按钮执行一次完整回测
- 展示 metrics
- 展示净值图和回撤图
- 展示订单表
- 展示 warning / error
- 继续输出 `orders/` 和 `reports/`

### 暂不支持
- 参数组合优化
- 历史回测任务列表
- 多策略切换
- 在线编辑股票池
- 登录、权限、数据库
- 异步任务队列
- 前后端分离部署

## 参数设计

第一版仅开放以下 4 个参数：

1. **回测引擎**
   - 可选：`vectorized` / `backtrader`
   - 直接复用现有回测能力

2. **动量窗口 `momentum_window`**
   - 影响信号排序，是最核心的策略参数

3. **持仓数 `max_holdings`**
   - 决定组合集中度，影响收益、回撤、换手

4. **初始资金 `initial_cash`**
   - 影响资金曲线展示和订单金额，可视化价值高

### 参数边界
- `engine`：仅允许 `vectorized` / `backtrader`
- `momentum_window`：正整数，建议页面限制在 `5..250`
- `max_holdings`：`1..len(SYMBOLS)`
- `initial_cash`：大于 `0`

### 默认值来源
全部默认值继续来自 `src/quant_a/config.py`。页面只覆盖本次运行值，不修改默认配置。

## 页面结构

### 侧边栏
- 回测引擎选择框
- 动量窗口输入控件
- 持仓数输入控件
- 初始资金输入控件
- “开始回测”按钮

### 主区域
1. **标题与说明区**
   - 页面名称
   - 一句策略说明

2. **运行摘要区**
   - 本次运行参数
   - 使用的引擎
   - 数据抓取 / 因子分析 warning

3. **指标区**
   - 年化收益
   - 夏普
   - 最大回撤
   - 波动率
   - 累计收益

4. **图表区**
   - 净值曲线
   - 回撤曲线
   - 如果现有报告中已有持仓图，可一起展示

5. **表格区**
   - 最新订单表
   - 可选展示最后一期目标权重摘要

6. **输出文件区**
   - `order_path`
   - `report_paths`

## 交互流程

1. 页面启动时加载默认配置，控件显示默认值。
2. 用户修改参数后点击“开始回测”。
3. 页面组装一次运行参数并调用统一入口。
4. 页面显示运行中状态。
5. 回测结束后按固定顺序展示摘要、指标、图表、订单和文件路径。
6. 如果发生错误，则停止结果渲染并显示错误信息。

### 交互约束
- 仅按钮触发执行，不做参数变化自动重跑
- 页面刷新后应保留最近一次控件值
- 页面必须明确显示运行状态

## 代码落点

### 1. 新增页面入口
新增根目录页面文件，例如 `streamlit_app.py`。

职责：
- 读取默认参数
- 提供控件
- 触发回测
- 展示结果

不负责：
- 策略逻辑
- 回测逻辑
- 数据抓取细节

### 2. 扩展统一入口
扩展 `src/quant_a/main.py` 的 `run_pipeline()`，让它除 `engine` 外，还可接受少量 runtime 参数覆盖。

建议支持：
- `momentum_window`
- `max_holdings`
- `initial_cash`

该入口仍然是 CLI、Notebook、Web 的统一编排层。

### 3. 扩展策略参数注入
在 `src/quant_a/strategy.py` 中让信号计算和目标权重构建函数支持可选参数传入，而不是只依赖配置常量。

影响范围：
- 动量计算
- Top N 选择

保持默认行为：
- 未传入时仍使用 `config.py` 默认值

### 4. 扩展回测参数注入
在 `src/quant_a/backtest.py` 中让回测函数支持 `initial_cash` 覆盖。

保持默认行为：
- 未传入时仍使用 `config.py` 默认值

### 5. 依赖调整
在 `pyproject.toml` 中新增 `streamlit` 依赖。

## 数据流

页面层不直接操作中间数据文件，而是调用统一入口获取结果字典。推荐继续沿用 `run_pipeline()` 当前返回 schema，并尽量保持稳定：

- `engine`
- `metrics`
- `backtest`
- `orders`
- `order_path`
- `report_paths`
- `warnings`
- `factor_analysis`

页面只消费这些结果，不重复拼装业务含义。

## 错误与 Warning 处理

### Warning
以下情况展示提示，但允许继续展示结果：
- live 抓数失败但本地缓存可用
- 因子分析样本不足或被跳过

### Error
以下情况直接展示错误并停止结果区渲染：
- 无缓存且抓数失败
- 不支持的回测引擎
- 回测没有生成有效日频记录
- 参数值超出允许边界

### 第一版不做
- 错误分类体系
- 自动重试
- 任务日志页

## 测试与验收

### 烟测
1. 用默认参数从页面成功跑通一次。
2. `vectorized` 与 `backtrader` 都能出结果。
3. 改动 `momentum_window` 后，结果发生变化。
4. 改动 `max_holdings` 后，结果发生变化。
5. 改动 `initial_cash` 后，资金或订单展示符合预期。
6. 页面可以显示 warning。
7. `orders/` 与 `reports/` 仍正常输出。

### 代码级验证
如果补少量自动化检查，优先验证：
- runtime 参数覆盖是否生效
- 未传覆盖参数时默认行为是否保持不变

## 非目标

以下内容明确不属于本次实现：
- 把项目改造成通用量化平台
- 在首版中抽象完整配置系统
- 为 Web 页面重构全部模块边界
- 引入 API 层、数据库、用户系统
- 处理公网部署和并发任务

## 实施建议

按以下顺序实施最稳妥：
1. 给统一入口补 runtime 参数覆盖
2. 给策略层与回测层补参数注入点
3. 新增 Streamlit 页面文件
4. 接入结果展示与 warning / error 展示
5. 做主流程烟测

## 成功标准

完成后，用户应能在不修改代码的情况下：
- 打开页面
- 调整 4 个核心参数
- 点击按钮运行策略
- 在线查看关键结果
- 同时继续使用项目已有文件输出

这意味着项目从“纯 CLI 研究脚手架”升级为“带轻交互界面的研究工具”，但仍保持最小复杂度。
